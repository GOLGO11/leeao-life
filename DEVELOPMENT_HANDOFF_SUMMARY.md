## 二、必须保留的硬约束（简版）

这些约束来自大交接文档，**不要在接手时丢掉**。

1. **必须一本一本、章节一段一段推进**，不要试图一次性扫完整套全集。
2. 年表目标是**大而全**，但必须仍然是“李敖相关事件”，不能把纯旁人背景硬塞进来。
3. 能精确到日就精确到日；不能精确到日时：
   - 用 `datePrecision`
   - 用 `displayDate` 保留原文日期形态
   - 不要伪造精确日
4. 同一年排序规则：
   - 先日级条目
   - 后模糊时间条目
5. 每条事件都要保留：
   - 书名
   - 章节
   - 文件路径
   - 转码后行号
6. 跨书同案或同链条，使用 `crossReferences`，不要乱挂。
7. 文本源文件大多是 **GB18030**，读正文请用：

   ```bash
   iconv -f GB18030 -t UTF-8 "<章节路径>" | nl -ba
   ```

8. 每轮新增后必须同步：
   - `data/ingestion-log.json`
   - `README.md`
   - `DEVELOPMENT_HANDOFF.md`
   - `exports/leeao-current-timeline.txt`
9. 如果新增了新的数据文件，必须同时更新：
   - `app.js` 的 `dataFiles`
   - `tools/export-timeline-text.mjs` 的 `dataFiles`

## 三、我这几轮总结出的避坑经验（重点）

### 1. 不要把“日期命中”误当“可入表事件”

最常见错误就是：看到文中有日期，就想直接入库。

实际上要先分清它是哪一类：

- **李敖本人写作/改订/发表/补记节点** → 高优先，可入
- **文章后记中引用的外部回应** → 视是否能形成“回响线”决定
- **纯历史背景日期**（如法律沿革、清代旧事） → 一般不直接入

例子：

- `018.修改《医师法》与废止中医` 里有很多 1913、1929、1943、1948 等日期，**这些不能机械入表**；
- 但 `1962-11-01` 原登《文星》、`1963-01-01` 在《今日医药》发表李敖意见，这些是**李敖本人作品传播节点**，可以入。

### 2. 《传统下的独白》最有效的挖法，不是“全章扫完”，而是“作品史分层”

前三轮已经证明，这本书最好的推进方式是按层次抽：

- 文末署日
- 写作地 / 写作完成节点
- 原刊发表节点
- 后来改订节点
- 后记里对发表后反响、文祸、被骂、被告的补记

换句话说，**先把作品史骨架立起来，再往外补厚度**，效率远高于逐段抄录。

### 3. 同一篇文章里经常有多层日期，不要只取一个

有些章不是“一个日期 = 一个条目”，而是天然就带链条：

- 《充员官》：`1961-04-03` 原刊 → `1963-10-08` 改订
- 《假如我是女人》：`1961-07-07` 初稿 → `1963-06-01` 发表 → `1963-08-16` 后记补记

这类文章如果只取一个署日，会把作品史压扁。

### 4. 但也不要为了“多条”而过拆

反过来，后记里提到的每个旁支都不一定值得起条。

比如：

- 朋友送书
- 某人骂了作者
- 后记顺手举的历史例子

只有当它满足下面至少一条时，才值得考虑入表：

- 能构成李敖作品传播链的一环
- 能构成李敖的文祸 / 争论 / 被告链的一环
- 能补出此前缺失的重要写作现场、发表场域或改稿轨迹

### 5. “妇女节”“去年”“三年后”这类词，宁可保守也不要装作已校准

如果原文是：

- “1961年妇女节”
- “去年11月1号”
- “三年以后”

要先判断有没有足够稳定的换算依据。

目前项目里已经采用过一种可接受做法：

- `1961年妇女节` → 排序日期用 `1961-03-08`
- 但 `displayDate` 保留“1961年妇女节”
- `certainty` 标成 `inferred`

这种做法可以继续，但前提是**一定把原文形态保留下来**，不能伪装成原文直出公历日。

### 6. 导出文件第一次读取经常会看到旧表头，别被误导

实操里已经多次遇到：

- 先执行 `node tools/export-timeline-text.mjs`
- 紧接着立刻读 `exports/leeao-current-timeline.txt`
- 第一次读到的是旧 header

因此，正确做法是：

1. 先重建导出
2. 再读一次导出文件表头确认
3. 以**第二次确认结果**更新 handoff / 统计文案

不要因为第一次读到旧 header，就误判导出没更新。

## 六、接手时建议的最小工作流

1. 先看：

   ```bash
   git status --short
   ```

2. 再看当前摘要和大交接文档：

   - `DEVELOPMENT_HANDOFF_SUMMARY.md`
   - `DEVELOPMENT_HANDOFF.md`

3. 读取目标章节时，用 GB18030 解码：

   ```bash
   iconv -f GB18030 -t UTF-8 "<章节路径>" | nl -ba
   ```

4. 先确认是否属于以下四类之一：

   - 写作 / 写定
   - 原刊发表
   - 改订 / 重写
   - 后记补记 / 发表后回响

5. 再做 repo 内去重，尤其查：

   - `data/timeline-events-traditional-monologue.json`
   - 相关旧书数据文件

6. 新增后同步：

   - `data/ingestion-log.json`
   - `README.md`
   - `DEVELOPMENT_HANDOFF.md`
   - `exports/leeao-current-timeline.txt`
   - 若有新数据文件，再同步 `app.js` / `tools/export-timeline-text.mjs`

7. 最后验证：

   ```bash
   node --check app.js
   node tools/export-timeline-text.mjs
   ```

   并重新读取导出文件表头确认总数和精度统计。
